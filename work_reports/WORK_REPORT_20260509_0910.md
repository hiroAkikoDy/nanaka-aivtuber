### Q1（△ 部分正解）

**回答:** モデルやシステムプロンプトの変更を簡単に行える。langchain-openai → langchain-anthropic のようにモデル差し替えが可能。

**採点:** モデル差し替えの容易さは正しいが、STEP3・STEP4を見据えた最大のメリットではない。

**解説:** 最大のメリットは **`ChatPromptTemplate` によって `{emotion}` や `{memory}` のような変数をプロンプトに動的に埋め込めること**。STEP1の `system_prompt.txt` は固定テキストだったが、STEP2でテンプレート化したことで `create_chat(question, emotion="嬉しい", memory="〇〇さんは前回クレソンについて質問した")` のように呼び出すだけでプロンプト全体が動的に変わる。これがSTEP3（感情管理）・STEP4（視聴者記憶）の基盤。

---

### Q2（○ 正解）

**回答:** templateの内容をllmに送っている。LCELでは `chain = prompt | model` のようにプロンプトとLLMを `|` でつなげて処理の連鎖（Chain）を実装する。

**採点:** 正解。

**補足:** `|` 演算子の処理は以下の2ステップ：
1. `prompt_template.invoke({...})` → 変数穴埋め → 完成したメッセージリスト
2. その結果を `llm.invoke()` に渡す → LLM API呼び出し

STEP3以降で `| output_parser` や `| retriever` を追加する際も同じ記法で拡張可能。

---

### Q3（△ 部分正解）

**回答:**
- `("system", ...)` → LLMに送るシステムプロンプト（キャラクター設定） ✅
- `("placeholder", "{chat_history}")` → チャット画面に記録されるチャット履歴 ❌
- `("human", "{input}")` → ユーザーの入力テキスト ✅

**採点:** `placeholder` の説明が誤り。

**解説:** `placeholder` は LangChain の特殊なタプル型で、`{chat_history}` の位置に `HumanMessage`/`AIMessage` のリストを**そのまま展開**してLLMに送信する。チャット画面の「表示」とは無関係。

```
展開前: [SystemMessage(...)]  ← placeholderの位置に何もない
展開後: [SystemMessage(...), HumanMessage("こんにちは"), AIMessage("やあ！"), HumanMessage("元気？")]
```

これがLLMに送信されるメッセージ列全体。

### Q4（✗ 不正解）

**回答:** モジュール化することでその後の変更や回収のときに個別のコードを触らずに一度に変更ができる。

**採点:** PromptTemplateの一般的なメリットを回答しているが、質問の核心（なぜメッセージ型に変えたか）に答えていない。

**解説:** 移行理由は **`ChatPromptTemplate` の `placeholder` が要求する型が `BaseMessage`（HumanMessage/AIMessage）だから**。

```python
# STEP1: dict のリスト
self.chat_log.append({"question": "...", "answer": "..."})

# STEP2: LangChainのメッセージ型（placeholderが要求する型）
history.append(HumanMessage(content=chat["question"]))
history.append(AIMessage(content=chat["answer"]))
```

`placeholder` に dict のリストを渡すと型エラーになる。LangChainのテンプレートエンジンが HumanMessage/AIMessage を要求するため、内部データ構造を合わせた。

---

### Q5（✗ 不正解）

**回答:** voicevoxの2回のPOSTで音声に感情表現や抑揚をつけるパラメータを emotion や memory で操作する。

**採点:** emotion/memory は VOICEVOX とは無関係。全く別の層（プロンプト層）で機能する。

**解説:** 実際の処理フロー：

```
create_chat(question, emotion="嬉しい", memory="〇〇さん前回ありがとう")
        ↓
self.chain.invoke({
    "emotion": "嬉しい",
    "memory": "〇〇さん前回ありがとう",
    "chat_history": [...],
    "input": question
})
        ↓
ChatPromptTemplate が system_prompt_template.txt の
{emotion} → "嬉しい"
{memory} → "〇〇さん前回ありがとう"
に置換
        ↓
完成したシステムプロンプト（テキスト）がLLMに送信される
        ↓
LLMは「感情状態：嬉しい」を考慮して回答を生成
```

`emotion` も `memory` も **LLMへの指示（プロンプト）の一部** として機能する。VOICEVOXはその後の段階で、生成されたテキストを音声に変換するだけ。

レイヤー分離イメージ：
```
[プロンプト層]  emotion/memory → テンプレート変数 → LLMの回答が変わる
[音声層]        LLMの回答テキスト → VOICEVOX → 音声出力
```

STEP3では感情分析結果を `emotion` に渡し、STEP4ではNeo4jから取得した視聴者情報を `memory` に渡す。

## 全体まとめ

| 重要ポイント | 内容 |
|---|---|
| テンプレート変数 | `{emotion}`/`{memory}` でプロンプトを動的に変更（STEP3/4の基盤） |
| placeholder | LangChain メッセージ型（HumanMessage/AIMessage）のリストを展開する挿入ポイント |
| メッセージ型移行 | `placeholder` が BaseMessage を要求するため、dictから移行 |
| emotion/memory | LLMのプロンプトに注入される変数（VOICEVOXとは無関係） |

**Q4とQ5はSTEP3・STEP4で直接使う概念。実際に感情や記憶を注入するコードを書く際に再確認すること。**
