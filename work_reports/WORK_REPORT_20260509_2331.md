### Q1（○ 正解）

**回答:** LangGraphのワークフローで実行される各ノードによって更新された値を保存する仕組み。次のノードに渡すデータ構造だからTypedDictを使う。

**採点:** 正解。

**補足:** TypedDictを使うもう一つの利点は、LangGraphがステートのキーと型を静的に検証できるため、ノード間でデータが壊れたときに早期発見できる点。`dict` だと `state["emotoin"]` のようなタイポが実行時まで気づけない。

### Q3（✗ 不正解）

**回答:** 条件分岐エッジをエントリーポイントと混同。

**採点:** `add_conditional_edges` と `set_entry_point` が混同されている。

**解説:**

| 関数 | 役割 |
|---|---|
| `set_entry_point("node")` | グラフの**開始ノード**を指定する |
| `add_edge(A, B)` | AからBへ**常に**遷移する固定エッジ |
| `add_conditional_edges(A, func, map)` | `func` の戻り値によって**遷移先を分岐**させる |

AIVTuberでの例え：

```python
# 固定エッジ：コメント受信後は必ず感情分析へ
graph.add_edge("receive_comment", "analyze_emotion")

# 条件分岐：感情によって応答トーンを変える
graph.add_conditional_edges(
    "analyze_emotion",
    lambda state: state["emotion"],
    {
        "happy":  "respond_cheerful",
        "angry":  "respond_calm",
        "normal": "respond_normal",
    }
)
```

STEP3では感情状態によって遷移先を分ける際に `add_conditional_edges` が中核になる。

### Q4（△ 部分正解）

**回答:** チェックポインターのインスタンスを作成し、グラフのコンパイル時のオプションとして渡す。compile前はチェックポイントがない状態。

**採点:** チェックポインターの説明は正しいが、`compile()` の本来の主目的が抜けている。

**解説:** `compile()` がすることは2つ：

**① グラフの検証と実行可能オブジェクトへの変換（主目的）**
```python
app = graph.compile()
app.invoke({"comment": "こんにちは", "emotion": "normal"})
# compile()後でないとinvoke()できない
```

**② チェックポインターの注入（オプション）**
```python
from langgraph.checkpoint.memory import MemorySaver
app = graph.compile(checkpointer=MemorySaver())
```

チェックポインターはオプション引数であり、渡さなくてもコンパイル自体は成立する。「compile前はチェックポイントがない」ではなく、「compile前はグラフを実行できない」が本質。

### Q5（○ 正解）

**回答:** Chainよりも各機能がノード単位でモジュール化されているので、普段のプログラミングよりもモジュール化の意識が必要。

**採点:** 正解。

**補足:** 書籍が言う「難しさ」の本質：

- **従来の思考**: 「処理をどう順番に書くか」（手続き型）
- **LangGraphの思考**: 「どんな状態が存在し、何がその状態を変えるか」（状態遷移型）

AIVTuberでは「感情という状態があり、コメントというイベントがそれを変化させ、応答がその状態を読んで変わる」という設計。これはSTEP4のNeo4j視聴者記憶（「視聴者の訪問回数」という状態を永続化）にも自然につながる。

## 全体まとめ

| 重要ポイント | 内容 |
|---|---|
| TypedDict | ノード間のデータ構造定義 + 型検証によるタイポ防止 |
| ノード関数 | ステートを受け取り、更新差分のdictを返す |
| add_conditional_edges | 状態に応じて遷移先を動的に分岐（STEP3の中核） |

| 状態遷移思考 | 「状態をどう定義し、何が変えるか」から設計を始める |
